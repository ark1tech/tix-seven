System Architecture

Architectural Overview
	TixSeven is composed of four major components: the attendee: the gate unit, the AWS-hosted verification server, and the MOSIP Testbed. The attendee is the human actor at both ends of the lifecycle, present at purchase with a PhilSys ID and present again at the gate with the same credential. The gate unit is the embedded hardware installed at the venue, responsible for capturing the credential and actuating the physical barrier. The verification server, hosted in AWS, is the layer mediating all communication, enforcing business logic, and owning the full transaction lifecycle. The MOSIP Testbed is the external identity authority; the only component in the system that can validate a PhilSys QR payload and return a verified identity token.

Operational Phase Workflows
TixSeven’s operational lifecycle is divided into four phases and one administrative phase, each with a defined owner and a defined place in the sequence of a live event. Phase 0 occurs at the point of ticket purchase, which may be days or weeks before the event itself. Phases 1, 2, and 3 occur in immediate succession at the gate on the day of the event, forming a single end-to-end pipeline that begins with an attendee presenting their PhilSys ID and ends with a physical gate opening or holding closed. Phase 4 occurs after the event concludes, when the system discharges its obligation to minimize any retained identity-linked data. 
Phase 0: Ticket Issuance
Phase 0 is the root of the entire verification chain. Its purpose, aside from recording that a ticket was purchased, is to establish the identity binding that makes every subsequent phase meaningful in nature. The EventTicketLink record created in this phase is the sole artifact connecting the identity of the attendee to their entitlement to enter a specific event, and without it, Phase 2 has nothing to verify against. Everything that follows depends on this phase completing correctly.
The attendee initiates a ticket purchase through a ticket vendor or event organizer, either online or onsite, and is prompted to present their PhilSys National ID.
The ticketing platform extracts the UIN and demographic data embedded in the QR code and forwards it to the MOSIP Testbed.
The MOSIP Testbed validates that the:
UIN exists and is currently active in the registry.
The associated identity attributes; that is, the name and dob, match the registry records.
If verification fails, the transaction is rejected and the attendee is notified that their PhilSys ID could not be verified. The process terminates here.
If verification succeeds, MOSIP returns a Partner-Specific User Token (PSUT) scoped to the specific ticketing platform.
The attendee completes their payment.
The system computes a link_hash = hash(PSUT + event_ID) and stores a new link in the EventTicketLink table referencing the generated hash along with the associated event_id. The PSUT itself is never persisted.
The system creates a corresponding Ticket record with a status of unused, referencing the stored link entry.
When Phase 0 concludes, the attendee carries no new artifact; that is, no ticket, code, or mobile credential that needs to be preserved. The ticket exists in the database and the credential that will redeem it is the PhilSys ID the attendee already had when they arrived. That continuity is the premise on which the entire anti-scalping property rests.
Phase 1: Scanning at the Gate
Phase 1 is the physical sensing layer of the system. It is brief and hardware-centric, producing exactly one output: a QR payload and a gate identifier transmitted to the server. The quality of everything that follows hinges entirely on the accuracy of what is captured in this stage.
The attendee arrives at the venue and presents their PhilSys ID to the GM861S scanner mounted at the gate.
The GM861S decodes the QR code and outputs the signed payload as a serial string to the ESP8266.
The ESP8266 reads the decoded string and transmits the payload along with its gate_id to the intermediary server through an encrypted HTTPS POST request.
The ESP8266 enters a processing state, waiting for a response from the server. The gate holds closed and provides no feedback to the attendee until a response arrives or a timeout elapses.
Once the payload is transmitted, the gate unit has completed its role for the scan attempt. It waits passively while the server takes full ownership of the decisioning process.
Phase 2: Centralized Validation and Decisioning
	Phase 2 is the intelligence layer of the system, and the most consequential. The server must resolve two independent tracks prior to reaching a decision. The first is the gate context: is it operational, and is it currently assigned to an event that accepts entries. The second is the identity and ticket context: is this a valid PhilSys credential, and does the identity behind it hold a valid unused ticket. These tracks are resolved sequentially instead of in parallel, as a gate that is offline or unassigned makes identity verification unneeded; there is no point of forwarding a QR payload to an external service when the gate context has already made the outcome uncertain. Also, this sequencing limits the exposure of the system to MOSIP calls in degraded operational conditions.
The intermediary server receives the payload and gate_id from the ESP8266. 
The server validates the gate_id and resolves the gate assignment:
The server first parses the gate_id as a UUID.
If the value is malformed or does not correspond to any known gate, the server records a denied result with the reason INVALID_GATE_ID and returns DENY. The process ends here.
The server checks whether the gate is operational.
If the gate is offline, the server records a denied result with the reason GATE_OFFLINE and returns DENY. The process ends here.
The server retrieves the currently active assignment of the gate.
If no active assignment exists, the server records a denied result with the reason INVALID_GATE_ASSIGNMENT and returns DENY. The process ends here.
The server retrieves the event associated with the assignment.
If the referenced event no longer exists, the server records a denied result with the reason EVENT_NOT_FOUND and returns DENY.               The process ends here.
The server verifies that the event is in a scannable state.
If the event has already concluded, the server records a denied result with the reason EVENT_CONCLUDED and returns DENY. The process ends here.
If the event has been cancelled, the server records a denied result     with the reason EVENT_CANCELLED and returns DENY. The process ends here.
The server forwards the payload to the MOSIP Testbed for identity validation:
If the MOSIP service is unreachable, the server records a denied result with the reason SERVER_TIMEOUT and returns DENY. The process ends here.
If verification fails because the credential is inactive or does not satisfy MOSIP validation requirements, the server records a denied result with the reason IDENTITY_NOT_VERIFIED and returns DENY. The process ends here.
If verification succeeds, MOSIP returns the PSUT for the identity.
Using the verified identity information and the resolved event_id, the server recomputes the linking_hash  =  HMAC-SHA256(pepper, "{psut}:{event_id}").
The server queries the EventTicketLink table using the recomputed hash and the resolved event_id: 
If no match is found, the server records a denied result with the reason LINK_NOT_FOUND and returns DENY. The process ends here.
If a match is found, the server gets the associated link_id.
The server retrieves the ticket associated with the link:
If no ticket exists for the link, the server records a denied result with the reason TICKET_NOT_FOUND and returns DENY. The process ends here.
The server snapshots the current ticket status for audit purposes.
If the ticket has already been consumed and is marked USED, the server records a denied result with the reason TICKET_ALREADY_USED and returns DENY. The process ends here.
The server attempts to consume the ticket and finalize the entry of the attendee into the event:
The ticket is marked USED, and the server records the exact time it was consumed. Prior to doing so, however, the server checks one final time that the ticket has not already been claimed by another scan request. This safeguard is important in situations where two scans of the same ticket occur almost simultaneously, such as when an attendee attempts to reuse a QR at multiple gates or when duplicate requests are thrown in quick succession.
If another request has already marked the ticket as USED, the server treats the ticket as already consumed, records a denied result with the reason TICKET_ALREADY_USED, and returns DENY. The process ends here.
If the ticket is still valid and unused, the server successfully consumes it, records a granted result in the audit log, and returns GRANT to the ESP8266.
Regardless of whether access is granted or denied, the server records the scan attempt in an immutable audit log. Each log entry preserves the state of the transaction at the moment the verification process concluded, including the raw and parsed gate identifiers, snapshots of the associated gate and event, assignment references, ticket references and ticket status snapshots, the final verification result, and the denial reason when applicable.
If an unhandled internal error occurs at any point in the pipeline:
The server immediately rolls back all uncommitted database changes to prevent the system from entering an inconsistent state.
The server records a denied result with the reason INTERNAL_SERVER_ERROR and returns DENY.
Any partially resolved ticket information is discarded before the log entry is written so that incomplete or misleading state is never preserved in the audit trail. 
Phase 3: Hardware Actuation and Feedback
Phase 3 is where the decision of the server becomes a physical outcome in the world. It is the phase that eliminates the human judgment layer: the servo motor acts directly on the server signal, with no staff member in the loop between the decision and the physical consequence. An attendee the system has denied cannot be admitted by persuading a person; they can only be admitted by overriding the hardware, which is a physical security problem of an entirely different kind.
The ESP8266 receives the response of the server.
If the command is GRANT:
The ESP8266 activates the green LED.
The MG996R servo rotates to 90 degrees, unlocking and opening the gate.
After a set amount of time, the servo returns to 0 degrees, closing the gate.
The LED resets to idle in preparation for the next scan.
If the command is DENY:
The ESP8266 flashes the red LED, and the servo holds at 0 degrees, keeping the gate locked.
The LED resets to idle. The process ends here.
In neither case does the gate communicate the reason for its decision to the attendee. The LED indicates only the outcome, while the specific denial reason is recorded exclusively in the server log. This serves as both a privacy property and a security property in that from a privacy perspective, the exact reason for denial is not exposed to surrounding bystanders. From a security perspective, an attacker probing the system cannot rely on the feedback of the gate to determine which specific validation check failed or to infer the conditions under which denial occurs. 
Phase 4: Cleanup
Phase 4 completes the privacy lifecycle of the system. The event has concluded, and the system now fulfills its obligation to minimize the identity-linked data it retains. The central question of this phase is not what the system still needs to preserve, since tickets and audit logs remain necessary for reconciliation and accountability, but rather what the system is obligated to delete now that the identity binding has fulfilled its purpose. 
After the event finishes and is transitioned to CONCLUDED, the system deletes all EventTicketLink records that are related to that event, thereby removing the linkage between the pseudonymous identity representation used during verification and the corresponding ticket issuance record. This deletion minimizes the persistence of data tied to attendees.
The Ticket records themselves are retained for audit and financial reconciliation. Their link_id is cleared to NULL on deletion of the link, making them permanently unassociable with any identity.
All Log records are unaffected.
The split retention model reflects a deliberate balance between privacy and auditability. The EventTicketLink records are deleted because they constitute the bridge between a pseudonymous identity representation and a ticket. Once those records are removed, the identity association is irreversibly severed. The Ticket records are retained because they serve as evidence that a valid entry occurred: that a ticket was issued and used for a specific event, at a specific gate, and at a specific time. The Log records are retained for the same reason, and additionally because they provide the forensic history of every scanning attempt processed by the system.
When Phase 4 completes, the database retains the full operational history of the event without retaining any information from which an attendee’s identity could be reconstructed. The system can know how many tickets were issued and used, at which gates, and at what times. But, it cannot determine, and cannot be made to reveal, whose PhilSys credentials were associated with those transactions. That is the intended terminal state of the TixSeven lifecycle, and Phase 4 is the mechanism that establishes it.
Gate Unit Architecture
The gate unit is the physical endpoint of the TixSeven pipeline. It is composed of three primary peripherals coordinated by a single microcontroller, with two auxiliary components for feedback and presence detection. The design is intentionally minimal: the unit captures, transmits, and actuates, but never decides. All decisioning authority resides on the verification server, ensuring that the security posture of the system cannot be subverted by tampering with the gate hardware in isolation.
ESP8266: The ESP8266 NodeMCU is the central microcontroller of the gate unit. It coordinates every other peripheral on the gate: it reads serial data from the GM861S scanner, drives the MG996R servo motor, toggles the status LEDs, samples the LDR sensor for presence detection, and maintains the encrypted HTTPS session with the verification server over Wi-Fi. The ESP8266 was selected because it bundles a microcontroller and an 802.11 b/g/n radio into a single inexpensive module, removing the need for a separate networking subsystem. Its role in the system is strictly that of an I/O bridge: it does not parse the QR payload, does not interpret the signed PhilSys credential, and does not evaluate ticket validity. Its only logical responsibilities are framing the scan request, validating the shape of the server response, and translating the result field into the corresponding physical output.
GM861S: The GM861S is the QR code scanner mounted at the face of the gate. It is configured in continuous-scan mode and communicates with the ESP8266 over a software UART at 9600 baud, with its TXD line wired to GPIO D5 and its RXD line to GPIO D6. When an attendee presents a PhilSys ID, the GM861S decodes the embedded QR code and emits the signed payload as an ASCII serial string terminated by a carriage return. The scanner performs no validation of the payload contents; it acts purely as an optical-to-serial transducer. Its outputs are forwarded without modification by the ESP8266 to the verification server.
MG996R: The MG996R is the servo motor responsible for the physical actuation of the gate. It is driven over GPIO D9 by the ESP8266 using the standard Servo library, and is held at a closed angle of 180 degrees in the idle state. On a grant decision, the firmware commands the servo to rotate to 0 degrees, opening the gate; once the ultrasonic sensor confirms that the attendee has crossed the threshold, the firmware returns the servo to 180 degrees, closing the gate. The MG996R was selected for its torque-to-cost ratio and its ability to hold position under load, which is essential for keeping the gate physically closed against passive pressure during a denial.
Status LEDs: A green LED on GPIO D8 and a red LED on GPIO D4 provide the only feedback surface exposed to the attendee. They communicate the outcome of the verification attempt — grant or deny — but never the reason. The deliberate opacity of this feedback channel is a security property of the system; an attacker probing the gate cannot infer which specific validation check failed.
LDR Sensor: A light-dependent resistor, wired to the ESP8266's analog input A0, allows the firmware to detect when an attendee has physically crossed the gate threshold. The LDR is positioned so that an attendee walking through the gate momentarily occludes the ambient light reaching its surface, producing a sharp transition in its analog reading. This is used after a grant decision to keep the gate open until the attendee has fully passed through, rather than relying on a fixed-duration timer that could be defeated by tailgating. The LDR replaces an earlier ultrasonic distance sensor in the prototype; it was preferred for being passive, silent, and immune to the acoustic reflections produced by adjacent gate units operating in close proximity.
Communication with the Verification Server: The gate unit communicates with the verification server exclusively over HTTPS, terminating TLS on the ESP8266 using the BearSSL stack. The server presents a self-signed X.509 certificate with a Subject Alternative Name for its public IP; the corresponding root certificate is compiled into the firmware as an X509List, pinning the trusted issuer at flash time. Because BearSSL validates certificate notBefore and notAfter dates against the local clock, the firmware synchronizes its system time over NTP against pool.ntp.org and time.nist.gov before opening any TLS socket; without this step, every handshake would fail at validation. On boot, the ESP8266 connects to the configured Wi-Fi network and reaches a ready state once both the wireless association and the NTP sync have completed. When a scan occurs, the firmware constructs a JSON document containing the raw qr_payload and the hard-coded gate_id, attaches an X-Gate-Api-Key header carrying the shared secret that authenticates the gate unit to the server, and issues a single POST to /verify. The server response is parsed as JSON; the firmware reads only the result field and, when applicable, the reason field for diagnostic logging. A response of grant or deny is converted directly into the corresponding physical outcome by the firmware. Any other response — a non-2xx HTTP status, a malformed JSON body, a TLS failure, or a request timeout — is treated as an implicit denial: the gate remains closed, no LED is illuminated, and the firmware returns to its idle state to await the next scan.
Translation of Decisions Into Physical Outcomes: A grant response causes the firmware to illuminate the green LED, rotate the servo to the open angle, and hold the gate open until the LDR reports that an attendee has entered and then cleared the threshold — first waiting for the analog reading to drop below an occlusion threshold as the attendee blocks the sensor, then waiting for it to rise back above a clearance threshold as the attendee moves on — after which a brief settling delay elapses and the servo returns to the closed angle and the LED is extinguished. A deny response causes the firmware to illuminate the red LED for two seconds while the servo remains at the closed angle, after which the LED is extinguished. In neither case is the denial reason rendered to the attendee; it is recorded only in the server-side audit log. After any decision is acted upon, the firmware enters a three-second scan cooldown during which incoming serial data from the scanner is drained and discarded, preventing duplicate submissions from a scanner that may continue to emit the same payload while a credential is still in view.
Gate Unit Firmware
The firmware is a single Arduino sketch flashed to the ESP8266 and structured around the standard setup-and-loop lifecycle of the Arduino runtime. It is intentionally thin: the entire program is a few hundred lines of code, with no persistent state beyond a cooldown timestamp and no decisioning logic beyond a string comparison on the server response.
Configuration and Secrets: Wi-Fi credentials, the verification server URL, the shared gate API key, and the gate UUID are compiled into the firmware as constants. The pinned root certificate is likewise compiled in as a PEM-encoded string literal. These values are baked into the binary at flash time rather than provisioned at runtime; a gate unit is therefore tied to its identity and trust anchor from the moment it is programmed, and any rotation of the API key or certificate requires reflashing the device.
Setup: On boot, the firmware initializes serial logging, configures the LED, servo, and scanner pins, and writes the servo to its closed angle so the gate is physically secured before any network activity begins. The LDR's analog input does not require explicit pin configuration and is sampled on demand during the grant path. It then opens the software UART to the GM861S at 9600 baud, associates with the configured Wi-Fi network, and blocks until WL_CONNECTED is reported. Once the network is up, the firmware initiates an NTP sync and blocks again until the local clock advances past a sentinel value, ensuring that any subsequent TLS handshake will be able to validate certificate dates. The gate enters its ready state only after all of these steps have completed; until then, scans are not processed.
Loop: The main loop is a non-blocking poll on the scanner serial port. When data is available, the firmware checks whether the scan cooldown has elapsed; if it has not, the incoming bytes are drained and discarded to prevent backlogged scans from accumulating. If the cooldown has elapsed, the firmware reads the buffered QR payload up to the next carriage return, trims whitespace, and, if the result is non-empty, invokes the verification routine. The cooldown timestamp is updated only after the verification call returns, so the next scan window opens once the prior decision has been fully acted upon.
Verification Routine: The verifyTicket function performs a layered sequence of operations against the verification server. It first re-checks the Wi-Fi association and logs the local IP and RSSI for diagnostic visibility. It then opens a BearSSL::WiFiClientSecure session with a five-second connect timeout, performs a TLS handshake against the pinned certificate, and on success issues the JSON POST to /verify with a ten-second request timeout. The response body is parsed with ArduinoJson into a static document; the firmware reads the result string and, on a grant, drives the servo and green LED while polling the LDR sensor to detect attendee crossing, or on a deny, illuminates the red LED for the denial dwell period. Any error path — connection failure, non-success HTTP status, JSON parse failure, or an unrecognized result value — is logged over the serial console and the gate is left in its closed, locked state. The firmware then closes the HTTPS session and returns control to the main loop.
Failure Posture: The firmware is fail-closed by construction. There is no code path in which the gate opens absent an explicit grant response from the verification server. Network failures, TLS failures, malformed responses, and unknown result values all collapse to the same physical outcome as an explicit denial. This property is what allows the gate hardware to remain a thin, untrusted endpoint: a compromised or malfunctioning unit can refuse entries that should have been granted, but it cannot, on its own authority, grant an entry that the server would have denied.

Backend Architecture
The TixSeven server is a FastAPI application that is backed by a PostgreSQL database. It serves as the system’s central orchestration and decision-making layer; it receives scan requests from gate units, mediates all communication with the MOSIP Testbed, enforces every business logic rule, and owns the transaction lifecycle for both ticket issuance and gate verification. The gate hardware itself remains intentionally thin, acting primarily as an input and display device that executes the decisions of the gate server.
The backend is organized into distinct architectural layers that separate external integrations, shared infrastructure, persistence logic, domain models, and business workflows:

Adapters
The adapter layer sits at the outermost boundary of the server, mediating all communication with systems that are outside the direct control of the server. In the current implementation, the only external system is the MOSIP Testbed.
The MOSIP adapter is defined through the MOSIPAdapter interface, which exposes a single verify operation. This abstraction isolates MOSIP-specific communication logic from the rest of the backend, allowing pipeline services to interact only with normalized verification results rather than with the MOSIP API directly. 
Two implementations of the interface exist.
RealMOSIPAdapter is the abstraction utilized when connected to the MOSIP Testbed. It is responsible for performing the external verification request and translating the resulting response into the backend’s internal identity model. Operational failures like timeouts, unreachable endpoints, or unusable responses are all surfaced as MOSIPUnavailableError, allowing the verification pipeline to handle infrastructure failures consistently. 
StubMOSIPAdapter supports local development and testing without MOSIP connection. Instead of performing network communication, it evaluates scanned payloads against locally defined mock verification data and returns development-mode verification results.
The adapter abstraction serves a purpose beyond testability. By isolating all MOSIP-specific protocol knowledge behind an interface, the server is insulated from changes in the MOSIP API. As a result, changes to the MOSIP endpoint, authentication mechanism,     QR payload format, or response structure can be handled inside the real adapter without forcing the service layer to change. 
Core
The core layer contains shared infrastructure that supports the rest of the server without belonging to any single architectural layer. It includes centralized configuration management, cryptographic utilities, JWT verification, trace handling, and reusable helper functions utilized in the service layer.
Configuration is managed through a settings object that loads values from environment variables and the server’s .env file at startup. The most security-critical settings here are the MOSIP connection parameters and the HMAC pepper used in the hash contract. The pepper functions as a server-side secret and must remain consistent across ticket issuance and verification so previously generated ticket bindings remain valid.
The cryptographic utility is centered around a single function, hash_psut(psut, event_id) -> str, which encapsulates the full hash contract of the system in one place. The function computes HMAC-SHA256(pepper, "psut:event_id"), ensuring that both ticket issuance and gate verification use the exact same computation without duplicating hashing logic across layers. 
The directory also contains a dedicated Supabase JWT verification module that is responsible for validating access tokens through JWKS-based ES256 signature verification, and a lightweight trace context utility that maintains a consistent trace identifier across the lifetime of a request for logging and observability purposes. Finally, the utilities module provides shared helper functions employed throughout the service layer.
Database
The db layer provides the foundational infrastructure needed for persistence and transaction management. It defines the declarative base class from which all ORM models inherit, configures the SQLAlchemy engine and session factory, and centralizes utilities such as server-side Philippine time defaults for timestamp columns.
Session management adheres to the standard FastAPI dependency injection pattern. A database session is created per request then injected into the repository and service layers, and closed automatically once the request completes. Sessions are not shared across concurrent requests and are never intended to persist beyond the lifetime of a single request.
Transaction boundaries are owned by the service layer instead of the repository layer. Repositories operate within the session and transaction context established by the service, allowing higher-level business operations to coordinate multiple repository calls within a single atomic transaction.
A seeder file also exists for testing.
Models
The models layer defines the SQLAlchemy ORM classes that map the domain entities of the application to the underlying database schema. Each model corresponds to a database table, including Venue, Event, Gate, GateAssignment, EventTicketLink, Ticket, and Log. The models define column structures, relationships, indexes, and database-level constraints, but intentionally avoid embedding business logic; operational rules and workflow decisions belong to the service layer.
Several of the constraint definitions are significant not only merely as schema details, but as core integrity and security mechanisms. The partial unique index on GateAssignment, which enforces that a gate may have at most one active assignment at a time, prevents conflicting event contexts from existing simultaneously at the database level. The consistency constraints on Ticket and GateAssignment ensure that lifecycle timestamps such as used_at and unassigned_at always align with their corresponding status values. Similarly, the constraints on Log enforce internal consistency between scan outcomes, denial reasons, and snapshot fields so that audit records remain structurally valid even under failure conditions.
These guarantees are enforced directly by the database rather than by the application layer alone. Hence, the invariants remain protected regardless of how the data is accessed, whether through the application, through a migration script, or through a direct database connection. The application enforces its rules; the database enforces the conditions that must never be contravened under any circumstance.
Schemas
schemas.py defines the Pydantic models that govern what the server accepts, returns, and carries internally during request processing. It handles request shape validation, response serialization, and pipeline context accumulation. The file does not make business decisions; it defines data shapes, boundary contracts, and lightweight validation rules.
The pipeline schemas deserve particular attention because they serve a dual purpose. VerifyContext and IssueContext are not ordinary request or response models; instead, they are accumulator objects populated incrementally as the verification or issuance pipeline progresses. A freshly initialized context contains only the original request inputs. By the conclusion of the pipeline, it contains the resolved gate, event, assignment, identity, ticket, response, and snapshot values touched during processing. This allows the final logging step to write a complete audit entry from a single self-contained context object without needing to re-query the same data.
The separation between EventUpdateRequest and EventStatusUpdateRequest also reflects a deliberate boundary constraint. General event field updates are kept separate from event lifecycle transitions so a caller cannot change the status of an event through the ordinary update payload. Because status is absent from the field-update schema, every status transition must pass through the dedicated status-transition path where the valid transition rules can be enforced by the service layer.
Repositories
The repositories directory provides structured, typed access to the database. Each repository receives a SQLAlchemy session as its constructor dependency and exposes focused query and persistence methods that can return ORM models, aggregates, primitives, or None when no matching row exists. Repositories simply execute database operations and return the result to the service layer; they do not contain business logic or interpret what missing data means. 
This separation has an important architectural consequence: since repositories are injected into services as dependencies, the verification and issuance pipelines can be tested through mock repository implementations without necessitating a live database connection. The most security-sensitive workflows in the system are therefore testable in isolation from the persistence layer.
The repositories are organized around the entities and workflows they support. VenueRepository provides basic venue lookup and existence operations through methods such as get_by_id, get_all, and exists. EventRepository extends beyond simple retrieval by exposing methods such as get_by_id_with_venue and get_all_with_venue, which eagerly load venue relationships to avoid repeated lazy-loading queries when event responses require venue information. It also provides aggregation-oriented methods such as get_assigned_gates, get_admitted_count, and get_admitted_counts_in_bulk, reflecting the fact that event detail views depend heavily on derived operational data rather than raw rows alone.
GateRepository exposes several distinct access patterns for active assignments, including get_active_assignment, get_active_event_id, and get_active_assignment _and_event_ids. These exist since different service paths require different levels of detail; some workflows need the full mutable assignment object while others require only identifiers for pipeline context accumulation. The same repository also owns assignment lifecycle persistence through create_assignment and deactivate_assignment.
TicketRepository supports the issuance and verification pipelines through methods such as find_link, find_ticket_by_link, create_link, create_ticket, and mark_used. The mark_used operation is particularly important because it performs the conditional update that acts as the verification pipeline’s concurrency guard, ensuring that only one verification request can successfully claim an unused ticket. The repository also provides reporting-oriented methods such as get_summary and get_all_tickets_by_event.
Finally, LogRepository encapsulates the audit and analytics queries used by the dashboard layer. get_summary, get_all_logs_by_event, and get_gate_filter_options focus primarily on aggregation, filtering, and historical retrieval rather than transactional workflow support.
Services
The services directory is where all business logic lives. It is responsible for interpreting requests, enforcing workflow rules, coordinating repository operations, and producing the final response. Nothing above the service layer makes business decisions, and nothing below it interprets them.
	The services divide into two architecturally distinct types that must not be conflated. IssuanceService and VerificationService are pipeline services that manage multi-stage request flows through mutable context objects. As the pipeline progresses, the context accumulates the resolved values required by later stages and by the final logging operation. Failures inside the pipeline are propagated through dedicated sentinel exceptions that carry both the denial reason and the accumulated context, enabling the outer handler to cleanly terminate the pipeline while still committing a log entry.
	These pipeline services also own their transaction boundaries explicitly. Controlled denials trigger a rollback of the in-progress transaction before the log is written and committed separately. Unhandled failures follow the same pattern and additionally return a generic error response as to ensure that operational logs are preserved regardless of how the request terminates.
EventService, GateService, VenueService, TicketService, and LogsService serve a different role. They are conventional CRUD-oriented services that perform request validation, repository coordination, and business-rule enforcement for ordinary dashboard operations. Their transaction flow is comparatively straightforward: a request is processed, committed, and returned without the staged control flow used by the pipeline services.
IdentityService occupies a position adjacent to both types. It is not a pipeline service in the full sense in that it does not manage a multi-stage context, but it is used by both pipeline services rather than being a CRUD handler. It encapsulates the identity-verification and link-hash operations shared between issuance and verification, ensuring that both pipelines use an identical MOSIP verification and hash-computation contract.
